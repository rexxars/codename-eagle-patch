# ce-patch technical details

How the patch is implemented. Most users only need the [README](../README.md); this is for anyone maintaining or auditing it.

The patcher (`patch.exe`) makes fixed-offset byte edits to two game binaries (`ce.exe`, `lobby.exe`), retunes one data file (`data4.bin`, the single-player weapon table), and installs three prebuilt binaries (`iplist.exe`, a patched `menudll.dll`, and `cemusic.dll`). It verifies the expected bytes at every byte-patch offset before writing, is idempotent, and backs up each file once. The bytes are generated and verified by `make-ce-patch.js` (assembled with `rasm2`, then round-trip disassembled); `src/ce_gen.rs` is its `--rust` output (do not hand-edit). Image bases: `ce.exe` / `LOBBY.EXE` = `0x400000`, `MENUDLL.DLL` = `0x10000000`; `file_offset = vaddr - imagebase`. Throughout this document, functions in the game binaries are referred to by their virtual address (e.g. "the version getter at `0x472060`") - drop the image base for the file offset, or feed the address to any disassembler to find the code. The patched sites are byte-identical across CE 1.41 / 1.42 / 1.43 with one deliberate exception: the version-getter immediate (fix #11) is `143` only on 1.43, so since the 1.50 protocol bump the ce.exe patch applies to **1.43 installs only** - on 1.41/1.42 the verify step refuses the file (those builds were never wire-compatible with 1.43 peers anyway; upgrade to 1.43 first).

## The nineteen fixes

### 1. Server browser finds internet + LAN servers (`iplist.exe`)

The stock in-game browser probes a static `iplist.txt` with a legacy protocol no modern server answers, so it stays empty. The replacement **`iplist.exe`** (built from the `iplist/` Rust project) is a drop-in for the prober the menu already spawns. The menu fires an unconditional `255.255.255.255` "LAN sweep" each time the player clicks **Refresh list** (it does _not_ auto-query on browser open - the list stays empty on first open, or shows the previous results, until Refresh is clicked); the new prober answers that by returning **both** the live community server list (fetched over HTTPS from the CEServers.net master list) **and** hosts discovered on the LAN, deduped against any manual `iplist.txt` entries. Per-server rows carry real name/map/players via a GameSpy `\status\` query. ce-patch installs `iplist.exe` over the stock one (backing the stock up once).

The in-game menu also needs a patch to _display_ those results: stock `MENUDLL.DLL` doesn't show map names and its list column is too narrow. ce-patch ships a **prebuilt patched `menudll.dll`** (map-name string display, a 15-char map column, a **25-char server-name column** - up from stock's 15, grown into the 20 blank columns that already sat between name and map - and a base-relocation fix that prevents a load-time crash from the patched code - see the notes in `menudll/make-menudll-patch.js`) and installs it like `iplist.exe`. The `menu.log` redirect (below) is baked into that same binary.

### 2. Hosted games get announced (master-server redirect, ce.exe)

CE hard-codes its GameSpy master hostname as `master.gamespy.com`, which is dead. A plain string swap replaces it with `ceservers.net` (NUL-padded in place; same length-or-shorter, no offset changes), where a community master listens for the game's `cneagle` heartbeat (UDP 27900), so hosted games get listed again. The same swap mechanism also rewrites the dead `codenameeagle.com` in-game URLs (demo "visit us" links, the version-mismatch message) to `codenameeagle.net` (equal length, no padding needed).

### 3. `cneagle://` one-click join (ce.exe)

`ce.exe` becomes its own URL-protocol handler. A Windows protocol launch hands the process the whole URL as one argument (e.g. `"...\ce.exe" cneagle://1.2.3.4:5678/`) and starts it with a foreign working directory. The patch handles both: see the startup routine below. The scheme match is case-insensitive (`cneagle://`, `CNEAGLE://`, ...). Registration is per-user (`HKCU\Software\Classes\cneagle`), no admin needed.

### 4. Confine output to the game folder (logs, screenshots; ce.exe + lobby + menudll)

CE writes assorted files to the game directory and even to the `C:\` root (`c:\error.log`, `c:\slave.log`, `c:\menu.log`), which modern non-admin Windows can block. The patch:

- `chdir`s `ce.exe` to its own folder on startup and creates `logs\` + `screenshots\` (see the startup routine);
- redirects every log to `logs\` and screenshots to `screenshots\` by repointing the filename strings: `error`/`slave`/`host`/`chat` (ce.exe), `lobby.log` (LOBBY.EXE) and `menu.log` (MENUDLL.DLL). (Screenshots also land in `screenshots\`, but their names are built by the fix-#19 naming stub rather than a repoint - the stock `shot%d.tga` push sits inside the region that stub displaces.)
- **roots the registry "Drive" at the install folder** instead of a CD letter. CE reads `HKLM\Software\Take2\Codename Eagle` value `Drive` (via the getter at `0x44c120`) and prepends it to asset/cutscene paths (e.g. `24bits\`, `cutscn\`). The original CD installer leaves that machine-wide value pointing at the **CD drive letter**; with the CD mounted, CE roots asset loads at the CD, the texture load returns NULL, and an unchecked virtual call crashes (`0x460fb1`). The patch stubs the getter to return a buffer (`s_gamedir` in the `.data` routine) that the startup routine fills with `<cedir>`, so every `Drive`-prefixed path resolves under the install folder regardless of any stale `Drive`/CD. This also makes **cutscenes work with the CD unmounted**: CE builds cutscene paths as `<Drive>\cutscn\<name>`, so copying the CD's `CUTSCN` folder into `<cedir>\cutscn` lets them play locally (full game only; the demo has no cutscenes).

### 5. No diagnostic clutter (`player<N>.txt`, LOBBY.EXE)

`lobby.exe` dumps a `player<N>.txt` connection-quality report on exit (two code paths). Both are suppressed by jumping over the dump block (`sprintf` name -> `fopen` -> `fprintf` -> `fclose`), leaving the rest of each function intact.

### 6. Music without the CD (`cemusic.dll`)

CE plays music as **CD Redbook audio** via MCI `cdaudio` (play at `0x483020` / stop at `0x483170` / loop at `0x4831d0`), with volume on the mixer CD line. With no CD it is silent. ce-patch ships a small 32-bit Rust DLL, **`cemusic.dll`** (rodio + a pure-Rust Vorbis decoder), that plays `<gamedir>\music\trackNN.ogg` (patent-free Ogg Vorbis; `NN` = the CD track number) on a loop instead.

The startup routine `LoadLibraryA`s it and resolves three exports (`cemusic_play`/`cemusic_stop`/`cemusic_volume`) into `.data` slots; the play/stop functions and the **CD/music** volume setter at `0x486b10` (not the shared music+SFX one) are **detoured** into cave stubs that forward to those exports.

Because playback runs on the DLL's own output stream and rodio's `Sink::set_volume` is a software gain, the **in-game music slider controls only music** (the SFX run on a separate DirectSound stream and are untouched) - the volume detour forwards the engine's existing music-volume fraction to it. A missing track file just plays silence; if `cemusic.dll` itself is absent the slots stay null and the stubs run the original code, i.e. CD music as before (purely additive). See [`cemusic/README.md`](../cemusic/README.md).

### 7. Single-player crash fixes (ce.exe)

Two long-standing single-player crashes, both fatal, both fixed with fixed-offset `ce.exe` edits (byte-identical across 1.41/1.42/1.43). These are separate from the multiplayer work but ride along in the same patcher.

**(a) The floating-point crash near enemy bases.** Approaching a base (pillbox / bunker / armored-car cannon emplacement) throws a `floating point error` and crashes. Root cause: 1.41 rewrote the mounted-turret **barrel-positioning** code. Where 1.36 placed the barrel at _fixed_ offsets, 1.41 computes it dynamically as `word[owner+0x2C0] / (k * [owner+0x6C])` in new per-part handlers (`0x403990`, `0x40fb00`, `0x40fb40` - reached only through the object's `+0x310` update pointer, so static analysis easily misses them). `[owner+0x6C]` is a per-emplacement value the constructor sets to 5 - **but it is 0 on a turret that was rebuilt by a savegame load** (see below), so the handler does `0 / 0 = NaN`. The NaN becomes the barrel's Z, spreads through the `pos = M*pos` transform (`0x46c550`) to the turret's world position, and then reaches `acos()` in the aim math (`0x425350`) -> FP domain error, plus a stack overflow from the NaN propagating through the AI recursion. Rather than chase the first NaN write through the object graph, the fix guards the two **consumption** points (robust, and inert on finite values):

- **A - position sanitizer.** Detour the `pos = M*pos` transform entry at `0x46c550` (9 bytes `d94204d94208d9420c` = the three `fld`s of `[edx+4/8/C]`) into a `.text`-tail cave at `0x4a0d96` that zeroes any inf/NaN of those three components (mask `& 0x7F800000 == 0x7F800000`) before the `fld`s, then jumps back to `0x46c559`.
- **B - acos guard (backstop).** Detour the aim's `fld [esp+8]; call acos` at `0x4253b3` into a cave at `0x4a0de8` that substitutes `1.0` for an inf/NaN input, runs `call 0x4943a0` (acos), and jumps back to `0x4253bc`.
- **Cave placement.** The caves start at `0x4a0d96`, not `0x4a0d95`: the original `.text` ends at VA `0x4a0d96`, and its final 6 bytes (`0x4a0d90`..`0x4a0d95`) are the import thunk `jmp [0x4a10ec]` = `KERNEL32!RtlUnwind`. A cave starting at `0x4a0d95` clips that thunk's last byte (`00` -> `50`, i.e. `jmp [0x504a10ec]`), which faults the instant the CRT unwinds a stack for **any** caught exception - a fatal `RtlUnwind` loop (e.g. crash-then-repeat at `0x4a0d90` when a cutscene raises an exception). The first free slack byte is `0x4a0d96`.

Both caves sit in `.text`-tail zero-fill; they are mapped/executable by bumping the `.text` section **VirtualSize** `0x9fd96 -> 0xa0000` (PE header field at file `0x220`). Because the guards only fire on inf/NaN, **freshly spawned turrets (which have `+0x6C = 5`, never go NaN) are completely unaffected and aim/track/fire normally** - the fix just stops the crash on the degenerate (reloaded) ones.

_Why reloaded turrets are broken (and why this isn't worth "fully" fixing):_ CE's savegame **load** rebuilds world objects with a generic object creator (`0x47ed20`), not the per-type constructors, so a reloaded turret is missing its constructor-set fields - the `+0x6C` divisor (=> the 1.41 crash) _and_ its AI/aim wiring (=> it sits inert). That stripped-on-reload behavior is pre-existing (1.36 does it too; 1.36 just never crashes because its barrel code has no divide), i.e. it is **not** a 1.41 regression and restoring it would mean re-running full construction on load. The 1.41-specific regression is only the crash, and that is what this fixes.

**(b) The save crash.** Saving builds a save-slot thumbnail by grabbing the screen (D3D device calls plus a GDI `StretchBlt`). The nine `E8` sites we NOP are **not** the grab itself - each one sits on a _failure branch_: the capture step runs (`call [edx+0x18]`, `call [ecx+0x44]`, `StretchBlt`, `call [edx+0x68]`, ...), then `test eax,eax; je <ok>`, and only if the step reports failure does the stock code `push` a label (`"D3DGrabScreen() 1"`, `"StretchBlt fails"`, ...) and `call` the fatal-error message routine at `0x4452d0` (`FatalError` in the RE names map). That diagnostic routine is what faults during a save capture on modern setups. NOP all nine of those diagnostic calls (`0x4602ce`, `0x4602f9`, `0x460319`, `0x460372`, `0x460392`, `0x4603b3`, `0x4604c3`, `0x4604ee`, `0x46050e`; each `E8 rel32` -> `90*5`) and the crash is gone without touching the capture: when the grab succeeds (e.g. under the bundled dgVoodoo) the thumbnail is produced normally, and a genuinely failed step just proceeds quietly instead of crashing.

(The CD-insert prompt for cutscenes is already handled by fix #4: the registry-Drive stub roots asset paths at the install dir, so copying the CD's `CUTSCN` folder into `<gamedir>\cutscn` plays cutscenes locally - no separate edit needed.)

### 8. Single-player turret balance restored to v1.36 (`data4.bin`)

This edits a **data** file, not a binary, and is **single-player only** - it never touches multiplayer. Applied by `patch.exe` automatically (function `patch_data4` in `src/main.rs`).

Single player never really worked right after 1.36 here: 1.41 repointed the mounted car-cannon (`car2can.scr`, used by the pillbox / bunker / armored-car / BPTur turrets) from weapon **class 3** to **class 15**, which was a **multiplayer** balance change that also leaked into single player. Two SP consequences:

- Stock SP `data4.bin` defines only classes 0-13, so class 15 (and 14, the heli) are **undefined** in single player -> the turret's gun is degenerate (and fed the base floating-point crash that fix #7a defuses). `data4.bin` and `mdata4.bin` share their first 14 weapons and the same +0x78 byte-obfuscation, and stock SP `data4.bin` ends exactly where MP's class 14 begins, so we splice `mdata4.bin`'s class-14/15 tail onto the SP table (`sp + mp[len(sp):]`).
- Class 15's **stats** are a big multiplayer buff over the class-3 gun 1.36's SP turret used - **AmmoSpeed 150 vs 45, Damage 30 vs 10 (3x)** - making SP base turrets brutally lethal. We retune the **SP** class-15 entry back to the 1.36 single-player numbers with length-preserving text edits in the deobfuscated record (`AmmoSpeed:150.0`->`45.00`, `Damage:30.0`->`10.0`; both strings are unique to the class-15 record), then re-obfuscate.

**Multiplayer is unaffected:** MP reads `mdata4.bin`, which we never write, so the armored car / torpedo boats / helicopter keep their exact 1.41 balance. (The helicopter is class 14 and the torpedo boat isn't `car2can` at all, so they never shared the turret's class anyway; only the armored car shares class 15, and only in the SP table.) `car2can.scr` is shared by both modes and is left untouched (editing it would change MP too). The patch only acts on the recognised stock/spliced tables (else it skips, never corrupting), is idempotent, backs up `data4.bin`, and is verified byte-identical to the hand-checked result from a stock 1.41 `data4.bin`.

### 9. Evict a stale/other CE session before starting one (ce.exe)

Multiplayer runs the session/transport in a separate helper process, `lobby.exe`, which owns the UDP socket (port `24711`), the DirectInput devices, and its end of the IPC pipes. Three ways this black-screens a new game:

- a **crashed run** leaves a hung/orphaned `lobby.exe` holding the socket;
- a **dedicated server or hung game** is still running on the same box, and its `lobby.exe` holds `24711` (two hosts can't share the port);
- either way, the lobby-spawn function (`0x485cb0`) always `_spawnl`s a **fresh** lobby, which then hits `bind fail!` -> `TCP/IP protocol not found!`, exits, and the game sits on a black screen waiting for a session that died.

The fix applies a **"newest session wins"** policy: immediately before the spawn, it terminates **every other CE process on the machine** - all `ce.exe` whose PID is not ours, and all `lobby.exe`. Killing the other `ce.exe` first matters: a live server/game would otherwise re-spawn its lobby and re-grab the port. This is safe because the lobby spawn's sole call site (`0x442360`, network-gated) runs **once at session start, before the first level loads - never mid-game** - so it can't kill the live session it belongs to, and single-player never reaches it.

Its 5-byte entry instruction `mov eax,[0x4c2d14]` (position-independent, so it re-runs verbatim) is detoured into a `.data`-cave stub that:

- resolves `CreateProcessA` / `WaitForSingleObject` / `CloseHandle` / `GetCurrentProcessId` / `Sleep` (kernel32) and `wsprintfA` (user32) via `GetProcAddress`,
- stamps our own PID into `taskkill /F /IM ce.exe /FI "PID ne <pid>"` (via `wsprintfA`) and runs it - killing every _other_ ce.exe but **not** the process running the hook,
- runs `taskkill /F /IM lobby.exe`, both windowless (`CREATE_NO_WINDOW`) and each `WaitForSingleObject`ed (3 s cap) so the terminations complete,
- `Sleep(500)` so the OS reclaims the freed UDP socket before the new lobby binds,
- restores all registers/flags (`pushad`/`pushfd`), runs the displaced `mov eax,[0x4c2d14]`, and jumps back to entry+5.

The two `taskkill` invocations share a cave subroutine `k_spawn` (cmd ptr in `ecx`). If nothing else is running, both `taskkill`s report "not found" and exit - harmless. The stub shares fix #6's executable-`.data` cave (ce.exe has its relocations stripped, so it always loads at its preferred base and relocating the displaced absolute `mov` is safe).

**Trade-off:** this deliberately evicts a dedicated server running on the _same_ box when you launch a game there. That combo doesn't work anyway (port conflict), and a dedicated-server host normally isn't also gaming on that machine, so "newest wins" is the right default; per-PID orphan detection was considered and rejected as needless complexity.

### 10. Dedicated-server idle-CPU throttle (ce.exe)

The engine's per-tick update function (`0x477640`) is driven by several free-running loops (the in-game run loop at `0x47a6e0`, the `WaitForLocalPlayer` join loop, and the host/wait loops around `0x47a900`). None of them has a frame limiter or a `Sleep`, so an **idle dedicated server pins a CPU core** (measured ~one full core doing nothing).

The fix throttles the tick loop only while the server is genuinely idle. The tick function's 6-byte prologue `push ebp; mov ebp,esp; and esp,-8` (position-independent, so it re-runs verbatim) is detoured into a `.data`-cave stub that:

- saves all registers/flags (`pushad`/`pushfd`),
- checks the dedicated-mode bit (`byte [0x53fa6c] & 1`); if clear, skips straight to the displaced prologue - so **every non-dedicated client is completely unaffected**,
- calls the engine's own player-count getter (`0x472160`, which already subtracts the dedicated host's slot - the same value it reports as `numplayers` to GameSpy) and skips the yield if it is `> 0`,
- otherwise yields the CPU for up to 50 ms (~20 Hz),
- restores registers/flags, runs the displaced prologue, and jumps back to entry+6.

The yield must not stall the network. The engine is single-threaded and services **all** UDP - game traffic _and_ GameSpy `\status\` queries - by non-blocking-polling one receive socket (`DAT_004be214`; it's the only socket the engine `recvfrom`s, drained by a `select` with a `{0,0}` timeout at `0x424200`). So a blind `Sleep(50)` blocks that poll for the whole interval: incoming `\status\` queries and join packets sit unanswered until it wakes, and the server's own `:210` LAN beacon (emitted from the same loop) stalls too - which is exactly what a browser sees as **variable 30-1200 ms pings and servers that intermittently don't appear at all**. So the yield is instead a **blocking `select(&{fd=DAT_004be214}, timeout=50ms)`** (WSOCK32, called through the IAT at `0x4a12cc`): it returns the instant a packet arrives, so the loop services it that tick (query pings back down to ~one tick), while an empty socket still yields for the full 50 ms. If the socket isn't up yet (`== INVALID_SOCKET`) it falls back to `Sleep(50)`.

The instant a player connects the count goes positive and the yield stops - full tick rate during actual play. Measured effect on the Wine/Docker dedicated server: idle CPU dropped from ~74% of a core to ~10%. (An earlier revision used a blind `Sleep(50)` here; it was measured as "query responses unchanged" but that under-sampled - a single leisurely probe lands in a good window, while a real browser refresh sending back-to-back probes exposes the starvation. The `select` rework fixes it without giving back the CPU win.) Shares the same executable-`.data` cave as fixes #6/#9. It is unconditional in the installer because it is inert unless `+dedicated` set the flag.

### 11. Version bump: "CEBETA v1.43" -> "CE v1.50" (ce.exe, display + wire protocol)

The game's version number comes from one getter at `0x472060` (`mov eax, 143; ret` - `b8 8f 00 00 00 c3` at file `0x72060`), whose int is multiplied by `0.01` and sprintf'd wherever a version is shown. It has **seven** callers: five display sites (the in-game `"CEBETA v%.2f"` banner @ `0x736d5`, the `"CE v%.2f"` string @ `0x3e612`, both version-mismatch messages @ `0x3dcc9` / `0x3e6f0`, and the GameSpy `\gamever\cneagle%.2f` status reply @ `0x23c9a`) and two **network** sites: the version word written into the outgoing join packet @ `0x773fb`, and the incoming equality check that rejects mismatched joins @ `0x7723c`.

Two edits: the `"CEBETA v%.2f"` format string is rewritten in place to `"CE v%.2f"` (shorter, NUL-padded), and the getter's immediate is bumped `143 -> 150` (`b8 8f 00 00 00` -> `b8 96 00 00 00`). Because every caller - display and network alike - reads the same getter, this is a **real protocol bump**: patched installs show v1.50, report `gamever cneagle1.50` to server browsers, and only interoperate with other patched installs. The split is deliberate - the patch set has grown well beyond cosmetics, and cross-version play with stock 1.43 is not something we can keep promising. An unpatched peer is rejected by the engine's own check with the stock `"Different versions. Host CE v1.50, your CE v1.43"` message (both sides format their own numbers, so the message is accurate on either end). This means **every server, including the Docker dedicated server, must be re-patched** or 1.50 clients cannot join it.

### 12. Savegames confined to `saves\` (ce.exe + menudll)

Stock CE drops its savegames loose in the install root: `sg<N>.dat` for the numbered menu slots and `temp.dat`, the level-start restart save. Same tidiness treatment as the logs/screenshots (fix #4):

- the startup routine also creates `saves\`;
- `ce.exe` builds every slot name from a single `"sg%d.dat"` format string (`0x4c2890`), used by three sites in the menu-command dispatcher (`0x441d80`) and delayed loader (`0x441d20`): save (command 5), load (command 6), and the delayed load. All three `push` operands are repointed at a `saves\sg%d.dat` string in the `.data` routine;
- the `temp.dat` **save** site (`mov ecx, 0x4c2884` before the save call at `0x47c030`) is repointed the same way at `saves\temp.dat`;
- the `temp.dat` **load** site cannot be repointed: the delayed loader builds the name with an inline 9-byte copy of the string into its stack buffer (three loads + three stores, 29 bytes of code), which caps the name at 8 chars. The copy is detoured into a `.data`-cave stub that stores the padded 16-byte `saves\temp.dat` into the same 0x40-byte buffer by immediates, then resumes;
- `MENUDLL.DLL` builds `sg%d.dat` itself for the save/load slot menu (its own `.rdata` copy of the string, `push`ed from 5 duplicated populate/action paths); the bundled prebuilt repoints all 5 at a `saves\sg%d.dat` string in the `.data` slack. All 5 operands already carry HIGHLOW relocs, which stay valid (the new value is still an in-module address);
- the slot **thumbnails** already live under `MENU\` (`MENU\sg_sg%d.tga`) and are left where both binaries expect them;
- `patch.exe` migrates any loose `sg<N>.dat` / `temp.dat` it finds in the install root into `saves\` (never overwriting an existing file there), so pre-patch saves stay loadable.

### 13. Weapon-switch fire-cooldown reset - the "8 trick" (ce.exe)

The well-known "8 trick": re-selecting the weapon you already hold (e.g. pressing `8` for the bazooka right after each shot, usually macroed to a mouse button) cancels the weapon's remaining fire delay, so slow weapons (bazooka, sniper rifle, grenade, gas) fire roughly twice as fast as intended - the ~1.2 s weapon take-up replaces the bazooka's ~2 s fire delay. Switching to another weapon and back does the same.

Root cause: fire rate is gated in exactly one place (the input applier's gate at `0x47ab20`), which suppresses fire while `now - player[+0x68]` (tick of the player's last shot - one shared timestamp per player, not per-weapon) is less than `player[+0x6c]` (the held weapon's `FireDelay` from `mdata4.bin`). Weapon selection (`0x4121f0`) enters a "taking up weapon" state even when re-selecting the held weapon - and while that state is active, the per-tick character update **zeroes the last-shot timestamp every tick** (`mov dword [esi+0x68], 0` at `0x414be8`). With the timestamp zeroed, the gate sees the last shot as infinitely long ago and the first trigger pull after take-up always fires: the switch state _replaces_ the fire delay instead of running alongside it.

**Part 1** NOPs that single 7-byte store: switching no longer wipes the shot clock. On its own, though, that made the _shared_ clock strict - the new weapon's full delay was charged against the old weapon's shot - which broke legitimate quick-switch flows. The worst casualty was the slot-7 explosives: placing a stick (class 13 `EXPPACK`, `FireDelay` 4 s SP / 3 s MP) stamps the clock, and the detonator (the `Detonat` project - weapon class **4**, the grenade class, 1.5-2 s) was then blocked against it. The gate vetoes by silently clearing the trigger bit while the hand animation (driven from the input elsewhere) still plays, so laying a stick right after selecting the weapon, or triggering the detonator right after placing a stick, _animated_ but did nothing. Every switch _to_ a slow weapon had a milder version of the same phantom block (pistol shot -> sniper: up to ~0.8 s of dead trigger after take-up).

**Part 2** fixes the model instead of the symptom: the cooldown gets an **owner**. The gate at `0x47ab20` is a self-contained 0x30-byte function (two callers - the player path `0x47acdf`, an AI path `0x4201c6` - no mid-function entries) that both checks the clock and commits `[char+0x68] = now` on an allowed shot. Its 5-byte entry is detoured into a `.data`-cave stub that reimplements it in full, with two additions:

- **Take-up passthrough.** Input runs _before_ the per-tick character update, and the update both flips the switch state (`[char+0x2c4]`; `0`/`3` = ready) to ready and consumes the surviving fire bit - so on the tick a take-up completes, the gate still _sees_ the switch state, yet its verdict is the one the real shot spawns from. Stock's part-1 zeroing implicitly encoded this: it erased the gate's commit on every take-up tick _except_ the completion tick (the update's completion branch at `0x414bf7` doesn't zero), so a fire attempt during the draw left no trace while a held trigger fired exactly once at draw end. The stub mirrors that distinction explicitly, using the update's own completion test (`now - [char+0x2c8]` vs the duration float `[char+0x2cc]`; `fcom`/`test ah,5`/`jp` as at `0x414bd1`): still mid-draw -> allow **without committing** clock or owner (a click during the draw, where no shot can spawn, must not start a cooldown - committing there charged a full fresh delay from the click, ~4 s instead of stock's ~2 s, with a ghost fire animation); take-up completing this tick -> full logic, commits included (allowing without committing there let the completion-tick spawn leave no record, and the next held-trigger tick fired again through the owner-mismatch path: a double shot). Stock also proves the passthrough safe for every character type: anything whose rate limiting works in stock (mounted guns etc.) must sit in states `0`/`3`, since the zeroing disabled rate limiting everywhere else.
- **The owner check.** A 1024-slot word table of "the project index (+1) of the item that last fired", keyed by the character's object index (`[char+0x7c] & 0x3ff`). The held item is resolved the same way the engine itself does it (`[char+0x98]` -> object table `[0x53fa98]` -> `[itemObj+0x2b4]`, the project index - the character update compares that against `GetProjectIndexByName("Detonat")` at `0x414cea`). If the held item differs from the owner entry, the shot is allowed without consulting the clock (a different weapon fired last - its cooldown isn't yours); on any allowed shot the owner entry is set to the held item. If the item can't be resolved (no item object, out-of-range index, null object table), the stub falls back to the stock shared-clock check and leaves the owner untouched.

The combined rule: **the fire delay binds only the weapon that fired it.** Every 8-trick variant dies - re-selecting or switching away and back transfers nothing (part 1 keeps the clock, part 2's owner still matches), and actually firing another weapon in between costs that weapon's take-up plus its shot, which exceeds every abusable delay. Meanwhile stick -> detonator, pistol -> sniper, and every other cross-weapon switch fires as soon as take-up completes, like stock - so the phantom animation-without-effect is gone from all switch flows. Stale owner entries (death/respawn/level change reusing an object index) are harmless: a fresh character's zeroed `[+0x68]` passes the stock check regardless of the owner comparison.

The fire gate and the owner table run inside every lockstep peer's simulation on sim-deterministic inputs (object indexes, project indexes, the tick counter), so all peers agree; mixed patched/unpatched lobbies would desync, which the 1.50 protocol bump (fix #11) already prevents: all 1.50 peers have the fix, and stock peers can't join.

### 14. Spontaneous explosion bug - SEB (ce.exe)

The infamous 1.41 regression: firing at the ground (especially high-speed rounds, especially while diving in a plane) sometimes made the **shooter** blow up on the spot, uncredited. The community's unofficial 1.42 never fixed the code - it hid the bug per-map by placing giant invisible `sebguard` planes under the multiplayer levels' terrain so stray bullets hit those first, which is why SEB still occurred occasionally on 1.42 (the planes don't cover everything), still occurred on custom maps, and still occurs in single-player (the SP levels never got sebguards).

Root cause: 1.41 added a vertical-band gate to the per-frame object-vs-landscape pass (`DetectCollisions`, `0x470c30`). The band is the min/max vertical coordinate over all land vertices (computed at land load in `0x46e210`, globals `0x4ce868`/`0x4ce864`). Objects beyond the band on the sky side skip terrain tests (a sane optimization); objects past the **deepest land vertex** hit a new out-of-world branch at `0x470d1e`. A projectile only gets there by tunneling: the swept segment-vs-face test misses thin/steep geometry at some spots, the bullet passes through the ground and keeps flying until it crosses the map's deepest vertex. The branch then silently removes **ownerless** objects (death timer + `RecomputeObjectFlags` - the correct cleanup), but for an **owned** projectile it instead does, at `0x470d42`:

```
mov  ecx, [objTable + [owner+0x7c]*4]  ; the OWNER's first child = the shooter's own body
mov  eax, [ecx+0x30c]                  ; its damage handler (DamageVehicle)
push 0                                  ; no contact point
push 0                                  ; no contact normal
push 0x453b8000                         ; damage = 3000.0f
xor  edx, edx                           ; no attacker
call eax
```

3000 damage to the shooter's own body with no attacker: instant uncredited death - exactly the reported symptom. Almost certainly a typo for detonating the stray projectile itself (its own index is `[esi]`; the owner's child is `[[esi+0x2ec]+0x7c]`). 1.36 had no band gate at all: tunneled shots flew on until their lifetime expired, harmlessly, which is why SEB appeared in 1.41. The location dependence also falls out: the trigger depth is global (the map's deepest vertex), so tunneling near deep terrain (Fever Valley) crosses it well within the bullet's lifetime, while tunneling under high ground usually doesn't.

The fix is one byte: the owner==0 test `je 0x470d54` at `0x470d26` becomes `jmp`, so every out-of-world projectile takes the existing silent-removal path, restoring the 1.36 outcome (and making the 1.42 sebguard planes redundant, including on maps that never had them). Projectile removal runs inside every lockstep peer's simulation, so mixed patched/unpatched lobbies would desync on what used to be a seb - covered by the 1.50 protocol bump (fix #11). The site is byte-identical on 1.41/1.42/1.43.

Because the engine is fixed, 1.50 also **removes the 1.42 sebguard shims** from the shipped MP levels (`game/common/level128`-`level134`): the sebguard placements are stripped from each level's `data1.bin` (80-byte record splice) and `world.dat` (5-line block removal), the sebguard-only `objects.dat` files that 1.42 added to level128-132 are deleted outright, and level133/134's own `objects.dat` archives are rebuilt without the sebguard project (level134's rebuild came out byte-identical to the stock pre-1.42 `LEVEL248/OBJECTS.DAT`, confirming the rebuild is faithful). Besides tidiness this is the proof of the fix: with no guard planes left, any seb on 1.50 would mean the engine fix is wrong. All peers must agree on level content for lockstep, which holds because every 1.50 install gets these files from the installer.

### 15. Startup cutscenes relocated (ce.exe)

At boot the menu function (`0x4415a0`) plays **three** Smacker videos back-to-back before showing the menu: the Refraction Games logo (`cutscn\logga.smk`), the campaign intro (`cutscn\intro.smk`) and mission 1's opening cutscene (`cutscn\m1c2.smk`). The three names are built as a NUL-terminated pointer array on the stack at function entry (`0x4415b8`..`0x4415d2`), prefixed with the Drive path (`%s\%s`) and handed to the Smacker list player (`0x447c60`). On a stock install nobody ever saw this: without the CD (or a copied `CUTSCN` folder) `SmackOpen` fails and each entry is skipped silently. But fix #4's Drive stub deliberately makes cutscenes load from `<cedir>\cutscn\`, so on a 1.50 install with the CD's cutscenes copied in, every launch replayed all three. Notably, `m1c2.smk` appears in **no level script** (mission cutscenes are normally played by `.scr` scripts via `REFShowCutscene`, e.g. level 9's `MissionReady()` plays `m9c3.smk`) - the boot reel was the only place it ever played, so mission 1's story setup was tied to sitting through the launch videos.

Two edits:

- **The boot reel is skipped.** The playback block is gated by `test esi,esi; je 0x441718` at `0x4416bd`; the `je` -> `jmp` (one byte, `74` -> `eb`) makes it unreachable. Safe: the skip path and the after-playback path converge on identical code (`mov ecx,[0x4c26cc]; call 0x4400c0`, rejoin the menu loop), and nothing else jumps into the block (verified by rel8/rel32 target scan). The in-menu "view intro"/"view credits" commands - menudll return codes 8/9, handled at `0x442f4e`/`0x442f64` - are separate code paths and still work.
- **The campaign opening plays when the campaign starts instead.** The menu->level transition insn `mov eax,[0x557510]` at `0x4439b9` (the lifecycle code that sets the running level `DAT_004c2d14 = selection & 0xff`; position-independent, re-run verbatim in the cave) is detoured into a `.data`-cave stub that, when **not dedicated** (`byte[0x53fa6c] & 1 == 0`), the **selected level is 1**, and it **hasn't already run this process** (a once-flag in cave `.data`, zero at every launch), plays `intro.smk` then `m1c2.smk` - preserving the original viewing order (backstory, then mission-1 setup) - before the level load begins. It calls the engine's own play-cutscene-by-name routine `0x44c1a0` (fastcall, `ecx` = bare filename): the exact routine `REFShowCutscene` (`0x48cdb0`) wraps and the hardcoded level-12 `credits.smk` site (`0x44485a`) calls, so path building (`<Drive>\cutscn\<name>`) and input-skip handling are stock behavior. Both name strings already exist in `.data` (`intro.smk` @ `0x4c2ce4`; `m1c2.smk` @ `0x4c279b`, the tail of `cutscn\m1c2.smk`), so no new strings are needed.

  **Missing-file behavior (stock CD-check).** `0x44c1a0` tests the file with an `fopen` (`0x468f50`) and, on failure, runs CE's stock CD gate: `MessageBoxA("Please insert CD", MB_OKCANCEL)` (`0x4861c0` - verified its only caller), then retry-loops the open. Both OK and Cancel just pump the message queue (`0x485290`) and retry, so without the file there is no way past the prompt - this is the original game's copy-protection behavior, unchanged, and the same prompt any missing mission cutscene raises through `REFShowCutscene` (e.g. level 9's `m9c3.smk`). The intended 1.50 workflow removes the _requirement_, not the check: copy the CD's `cutscn\` folder in and fix #4 resolves `<cedir>\cutscn\<name>` locally, so nothing prompts. The only difference from stock is the route to the two opening videos - stock reached `intro.smk`/`m1c2.smk` through the boot-reel list player (`0x447c60`), which logs and skips a failed `SmackOpen` silently, whereas the relocation sends them through `0x44c1a0`'s prompt path like every other cutscene - so a full install that starts the campaign without copying the cutscenes gets the stock prompt at the opening rather than a silent skip.

The once-flag means the opening plays on the first entry into level 1 per process: a mission-1 restart or a level-1 savegame load later in the session won't replay it. (Loading a level-1 savegame as the _first_ level of a session does play it once - the save-load path funnels through the same transition - a small trade-off for keeping the trigger simple.) The engine's "leave level" teardown re-execs `ce.exe`, so returning to the menu that way resets the flag - starting a fresh campaign after that shows the opening again, which matches the intent. Dedicated servers never play cutscenes (the gate is checked first). All sites verified byte-identical on 1.41/1.42/1.43.

### 16. Resolution-scaled aiming crosshair (ce.exe)

The on-foot aim reticle (`FUN_00438080`, an interface element) draws `Target.tga` - an **8x8** texture (orange cross) - as a quad centered on the aim point (`ecx`=x, `edx`=y). Its path-1 branch hardcodes the quad at a fixed **8 pixels** (`sub esi,4` / `sub edi,4`, corners at +/-4), so the crosshair is 8 screen pixels regardless of resolution: fine at the game's ~480p-era design resolution, nearly invisible at 1080p+. (The function's other branch draws `snipemod.tga`, the sniper scope, which already scales via the render dimensions - untouched.)

Fix: detour the 5-byte `mov eax,[0x4dde5c]` at `0x43814a` (the load of the cached `Target.tga` handle, exactly where path 1 begins) into a `.data`-cave stub that computes `half = renderHeight (0x5555b4) / 120`, clamped to `>= 4` (so it never drops below the stock 8px floor), rebuilds the same quad at `+/- half`, then runs the stock load-if-absent + draw and returns to the path-1 epilogue at `0x4381a5`. `half = 4` at 480p (identical to stock), `10` at 1200p (a 20px crosshair) - the reticle keeps its original apparent size at every resolution. The divisor is a single tunable (`CROSSHAIR_DIV` in `make-ce-patch.js`). Since the stock `Target.tga` is only 8×8 (and its 7×7 cross content is ~0.5 texel off-centre), it upscales blockily; 1.50 also ships a crisp, exactly-centred **32×32** `Target.tga` (see `game/full-overrides/README.md`) so the enlarged crosshair renders cleanly - delivered the same two ways as the SNIPEMOD override (fix #17): patched into a full-game install's `24bits/textures.dat` by `textool` at install time, shipped in the demo's prebuilt `texsec.dat`. Site byte-identical on 1.41/1.42/1.43.

### 17. Smooth sniper-scope overlay (32-bit texture override)

The sniper scope (`snipemod.tga`, drawn 4x mirrored around screen center by `FUN_004380b0`) ships as a **24-bit** texture, so its "transparency" is the engine's binary black color-key: the lens edge and the reticle's width steps are hard 1-bit cutouts that upscale into visible staircase pixels at modern resolutions. A **32-bit** texture instead renders from its real alpha channel (ARGB4444 at runtime, 16 levels, bilinear-filtered), so 1.50 ships a regenerated 32-bit SNIPEMOD: the lens arc is recovered as a fitted ellipse (the artist's 320x240 screen-space circle squeezed by the 4:3 aspect; 0.23px mean fit residual) and rendered with supersampled coverage AA, the reticle bars keep their measured stock geometry with AA edges and a short taper at the 3-to-1-texel width step, and the hairline thins toward the mirror seam (1.0 -> 0.6 texels) so the screen-center cross reads thinner at high resolutions. The four graduation ticks stay stock-exact and hard-edged on purpose. A plain texture upscale is not an option - the texture loader hard-asserts on sources over 256x256 (`LoadFromTGACacge TO3FXX > 256`).

Delivery differs per variant. **Demo** installs ride the engine's texture-archive precedence: the by-name lookup (`0x4641b0`) searches the level's `leveltex.bin`, then `24bits/texsec.dat`, then `24bits/textures.dat` - first match wins - so the demo deliverables ship a small two-entry `24bits/texsec.dat` (the new SNIPEMOD plus fix #16's 32×32 `Target.tga`, built by `scripts/build-demo-texsec.js`) that shadows the stock entries in the demo's trimmed `textures.dat` (the demo never had a texsec.dat, but the engine probes for it unconditionally). **Full-game** installs instead get `SNIPEMOD.tga` replaced directly inside their own 151 MB `24bits/textures.dat` by `textool` at install time (see the install-time texture patching section below), so no shadowing entry is involved. The texture is authored in `game/full-overrides/24bits/snipemod32.tga` and regenerated by `scripts/make-snipemod32.js` (which also documents the geometry recovery); no level or `ce.exe` bytes are involved.

### 18. Dedicated-server log file (ce.exe)

The dedicated server emits its operational events - `Server loaded ok`, and `Player '<name>' joined the game` / `left the game` / `lost connection` / `WORLD OUT OF SYNC` - through a single console printer, `FUN_00443740`, which formats the line and calls `WriteConsoleA(DAT_004de8b4, ...)`. `DAT_004de8b4` is the handle from the dedicated console's `AllocConsole` (`FUN_004436b0`). That output only exists in the console screen buffer: it can't be redirected (`>`/`|`/`tee` don't see `WriteConsoleA`), and on a **headless host** it is lost outright - under Wine (the Docker dedicated server) `AllocConsole` yields no usable handle, so `DAT_004de8b4` is NULL and every line is written to a null handle and discarded.

The patch detours that one `WriteConsoleA` call site so the formatted line is also appended to `logs\server.log`. The 6-byte `call ds:[WriteConsoleA]` at `0x4437ad` becomes `jmp` (+`nop`) into a `.data`-cave stub that, using the buffer/length already on the stack as the call's arguments, does `CreateFileA("logs\server.log", FILE_APPEND_DATA, share RW, OPEN_ALWAYS)` -> `WriteFile` -> `CloseHandle` (open/append/close per line - these events are infrequent), then runs the displaced `WriteConsoleA` and jumps back to `0x4437b3`, so a real console (native Windows) is unchanged. `logs\` already exists and is the cwd by the time any line is printed (the startup routine, fix #4). Only this printer is hooked; the interactive command prompt's echo (`FUN_004437c0`'s `Ok>`, a separate `WriteConsoleA`) is deliberately left out of the file. The cave uses `ce.exe`'s existing imports (`CreateFileA`/`WriteFile`/`CloseHandle`) directly - no `GetProcAddress` needed.

### 19. Screenshot overhaul: full resolution, timestamp names, F11 (ce.exe)

Every screen capture goes through one per-renderer routine reached via the function pointer `0x53fae4` - in the Direct3D/DirectDraw renderer that is `D3DGrabScreen` (`0x460180`), which captures at a hardcoded **640x480** regardless of the render resolution: it mallocs `640*480*3` bytes, creates a 640x480 offscreen surface, `SetStretchBltMode(HALFTONE)` + `StretchBlt`s the front buffer (rendered at `0x5555b0` x `0x5555b4`, width x height) down into it, converts to 24-bit and hands the buffer to the TGA writer (`0x4646c0`). So a PrintScreen `shot<N>.tga` was always a filtered downscale to 640x480. (The Glide/3dfx renderer's `3DFXScreenDump` at `0x4631c0` already captures at the render resolution; only the D3D path is affected.)

The routine takes two register args: `ecx` = the output filename, `edx` = a mode. PrintScreen (`0x4443eb`) passes `edx=0`; the savegame path (`0x441ebe`) passes `edx=1` to also produce the save/load-menu thumbnail (`MENU\sg\sg<N>.tga`), which it derives from the capture with `HalfScalePic` (`0x464770`) - a loop that repeatedly **halves** the image until it fits the target box, so only a 640x480 source lands exactly on the 80x60 the menu expects. The thumbnail path must therefore keep the stock capture size.

Fix: the 21-byte size+dims setup at `0x4601ab` (`push 0xe1000; mov [esp+0x14],640; mov [esp+0x18],480`) is detoured into a `.data`-cave stub. For `mode != 0` (the mode arg is in `esi` at the site) or a zero width/height it re-runs the displaced original (stock 640x480); for a PrintScreen shot it pushes `width*height*3` as the malloc size and stores the live render width/height into the same two stack slots. Everything downstream - the offscreen-surface dims, the `StretchBlt` destination rect (which becomes a 1:1 copy), the 24-bit conversion, the row flip and the TGA header - reads those two slots, so the whole pipeline follows. The stub then jumps back to the malloc call at `0x4601c0`. Site byte-identical on 1.41/1.42/1.43.

**Timestamp names.** The shot number is a zero-initialized stack local of the lifecycle function `FUN_004438c0`, incremented per shot (`0x4443cf`) and fed to `sprintf("shot%d.tga", n)` - and the TGA writer `fopen`s `"wb"` - so every ce.exe run restarts at `shot1.tga` and silently overwrites the previous session's screenshots one by one. The whole 34-byte name-build at `0x4443cf` (counter load/inc/store/push + the `sprintf` into the stack buffer + the grab-call arg setup, ending right before the `call [0x53fae4]` at `0x4443f1`) is detoured into a cave stub that names the shot by local time instead: `screenshots\shot-YYYYMMDD-HHMMSS-mmm.tga` - collision-free across sessions (milliseconds cover a double-press within one second) and sorted chronologically by name. The stub calls `GetLocalTime` (kernel32; ce.exe doesn't import it, so the startup routine resolves it into a cave slot - left null on failure), pushes the seven `SYSTEMTIME` WORD fields as `movzx`'d dwords, `sprintf`s (CRT, `0x494186`) into a cave name buffer, sets `edx=0`/`ecx=&name` and jumps to the grab call. If `GetLocalTime` went unresolved it falls back to the displaced stock counter naming (relocated in the cave, using the `screenshots\shot%d.tga` redirect string). The stock `push "shot%d.tga"` at `0x4443dd` sits inside the displaced region, so the fix-#4 filename repoint for screenshots is gone - the redirect lives in this stub's format strings.

**F11 alternate key.** Windows 11 (22H2+) binds PrintScreen to the Snipping Tool by default; its overlay steals focus and knocks the exclusive-fullscreen game out (minimize), often before the grab completes. The engine polls PrintScreen at three sites in the handler - the trigger check plus a two-step wait-for-release debounce - each a 7-byte `mov cl,0xb7; call 0x483770` (the key-down poll: `cl` = DIK code, returns `eax` = 0/0x80 from the DirectInput state array at `0x5523b8`). Each site becomes `call` (+2 `nop`) into a cave stub that returns PrintScreen **OR** F11 (DIK `0x57`), preserving the poll's return convention. Patching all three sites keeps the debounce correct for either key (the wait-for-release loop waits until neither is down), so holding F11 still takes exactly one screenshot. F12 is not usable: it opens the in-game console (poll at `0x43b421`, present at least since 1.33). F11 is verified free in both 1.41 and 1.43 - a scan of every `mov cl,<key>; call 0x483770` site and every direct state-array read finds no `0x57`. Sites byte-identical on 1.41/1.42/1.43.

## The ce.exe startup routine

ce.exe's `.text` has no free code cave, so the startup logic lives in a routine written into the unused zero-fill of the **`.data`** section (which is marked executable by flipping its section-header `Characteristics` to add `IMAGE_SCN_MEM_EXECUTE`). It is reached by **detouring the CRT-startup `GetCommandLineA` call** (`0x496987`) - which runs before `argv` is parsed and before `WinMain` opens any log - into the routine, which then returns the (rewritten) command-line pointer. The routine:

1. calls the real `GetCommandLineA`;
2. resolves `SetCurrentDirectoryA` / `CreateDirectoryA` / `RegCreateKeyExA` via `GetProcAddress` (ce.exe imports none of them; it does import `LoadLibraryA`/`GetProcAddress`);
3. builds `<cedir>` from `GetModuleFileNameA`, copies it into the `s_gamedir` buffer (the value the stubbed Drive getter returns - fix #4), and `SetCurrentDirectory`s to it (this is the fix for protocol launches, which start in `system32` - the original `lobby.exe not found` failure);
4. `CreateDirectory("logs")`, `CreateDirectory("screenshots")` and `CreateDirectory("saves")` (relative, now that cwd is the game dir);
5. `LoadLibraryA("cemusic.dll")` and resolves `cemusic_play`/`cemusic_stop`/`cemusic_volume` into three `.data` slots (left null if absent - fix #6);
6. registers `HKCU\Software\Classes\cneagle` -> `"<cedir>\ce.exe" %1`;
7. rewrites the command line in place: `cneagle://` -> `+connect ` (both 10 bytes, case-insensitive match by OR-ing scheme bytes with `0x20`), trailing `/` trimmed;
8. returns the command-line pointer.

Registers carried across the API calls use callee-saved `ebx`/`esi`/`edi` only (never `eax`/`ecx`/`edx`) - an earlier bug held a pointer in `edx`, which the intervening calls clobbered and caused a startup crash.

## The patcher

`patch.exe` (Rust, `x86_64-pc-windows-gnu`, pure `std`) drives a generic edit model: every change is `(file_offset, orig_hex, new_hex)`; an empty `orig` means the target is zero-fill slack (the `.data` routine and the redirect strings). Per file it classifies each edit as done / patchable / unexpected, refuses the file if anything is unexpected (wrong build), reports "already patched" if all done, else backs the file up once and applies. Only `ce.exe` and `lobby.exe` are byte-patched this way (`ce.exe` additionally gets the master pattern swap). The three bundled binaries (`iplist.exe`, `menudll.dll`, `cemusic.dll`) are then copied from beside the patcher into the install dir, backing up each existing file once (a bundled binary that isn't next to the patcher is skipped). Files are matched case-insensitively (Windows is case-insensitive anyway).

`make-ce-patch.js` is the source for the ce.exe/lobby.exe bytes: it assembles the ce.exe `.data` routine, lays out the slack strings, computes the detour + every repoint operand + the `.data` exec flag + the lobby edits, round-trip disassembles, and emits `--rust > src/ce_gen.rs` (which also lists the bundled binaries). The bundled `menudll.dll` is produced by `menudll/make-menudll-patch.js` (below).

## Install-time texture patching (`textool`)

Fixes #16 (crosshair) and #17 (sniper scope), plus the gas-mask HUD icon (an `INTERFC1.tga` atlas edit), replace **textures** rather than code. On full-game installs those replacements are not shipped as rebuilt archives: the patch installer bundles **`textool.exe`** (the [`textool/`](../textool/) crate - pure `std`, `x86_64-pc-windows-gnu`, compiled fresh at release time like `menuinfo-nick`, never committed) and runs it from `$PLUGINSDIR` at install time to patch the player's **own** `24bits` archives in place: `INTERFC1.tga` into `texsec.dat`, `SNIPEMOD.tga` and `Target.tga` into `textures.dat`. (The MP demo is different: its deliverables ship a tiny prebuilt `texsec.dat` whose entries win the archive-precedence search, see fix #17; the patch installer applies no texture step to demo-variant installs.)

**The archive container.** `24bits/textures.dat` and `24bits/texsec.dat` share one format, all integers little-endian: a u32 entry count, then a **fixed 2048-slot TOC** (regardless of count) of 17-byte records, then the blobs - canonically concatenated in TOC order from byte 34820 (= 4 + 2048×17). Each record is a 13-byte name field (NUL-terminated Latin-1, max 12 name bytes; bytes after the NUL are don't-care, canonical fill `0xCC`) followed by a u32 **absolute** blob offset - no length field: a blob runs to the next entry's offset, the last to EOF. A blob is a standard uncompressed true-color TGA minus its first 8 constant bytes (`00 00 02 00 00 00 00 00`). Stored pixel rows are **top-down behind a bottom-origin descriptor** (the engine reads rows in file order and ignores the origin bit), so authored replacements must already be in engine row order - cnetool's `pngToTga({topDown: true})` convention; `textool` stores its inputs verbatim (minus the prefix) and never reorients. `textool` reads leniently (shipped archives carry garbage padding and non-canonical blob placement; offsets are bounds-checked and must be monotonic) and writes canonically, mirroring cnetool's `buildTextureArchive`.

**Why install-time.** Two reasons. Size: the three replacement TGAs total ~0.5 MB, while shipping rebuilt archives would mean a 3.3 MB `texsec.dat` - and `textures.dat` is 151 MB, never shippable at all, which is why earlier revisions shipped SNIPEMOD/Target as _extra_ `texsec.dat` entries that shadowed the `textures.dat` originals through the lookup precedence. Ownership: patching in place edits only the three named entries, so any other texture the player modded survives, and the installer no longer _adds_ SNIPEMOD/Target shadow entries to `texsec.dat`. (Installs upgraded from the 1.50 betas keep the two entries those betas appended - the new installer preserves an existing `texsec.dat` and only patches `INTERFC1.tga` into it. Harmless while the pixels match the ones patched into `textures.dat`, but a future change to those textures would need the stale `texsec.dat` entries pruned, or they would shadow it.) One wrinkle: stock **1.0** installs have no `texsec.dat` at all, and it cannot be skipped there (14 of its 27 textures exist in no other archive), so the installer stages the pristine 1.43 `texsec.dat` and writes it **only when absent** before patching; 1.41/1.43 installs keep their own copy.

**Failure containment.** `textool set` validates every staged TGA up front (uncompressed true-color, 24/32-bit, square power-of-two ≤ 1024, exact length) and upserts them all in memory (name match is ASCII-case-insensitive against the staged file's basename, preserving the stored name - stock stores `TARGET.tga`, we stage `Target.tga`); only then does it rewrite the archive **atomically**: temp sibling file, `fsync`, rename over the original. On any error nothing is written and it exits 1 (2 for usage errors); the installer checks only the exit code (`nsExec::ExecToLog`) and treats a failure as a non-fatal warning - the player keeps their current textures. One deliberate trade-off: `textool.exe` (like `menuinfo-nick`) is built as a **64-bit** binary - 32-bit mingw-w64 doesn't link cleanly against Rust's precompiled std (see the comment in `textool/.cargo/config.toml`), and an install-time helper the installer spawns is not bound by the game's 32-bit arch. On a 32-bit Windows box the invocation simply fails and the installer continues with that same warning: those installs keep their stock textures (the scope/crosshair changes are cosmetic; the level-6 gas-mask inventory icon then renders the stock `INTERFC1.tga` atlas cell).
